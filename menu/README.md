# Menu

This folder contains the restaurant menu data and the script that converts it to a normalized, relational JSON format.

## Files

| File | Description |
|---|---|
| `scanned-menu.json` | Raw scanned version of the menu (source of truth) |
| `menu.json` | Output of `convert.py` — normalized, camelCase structure |
| `convert.py` | Conversion script (Python) |
| `convert.js` | Conversion script (Node.js, equivalent to `convert.py`) |
| `menu-schema.json` | JSON Schema (draft-07) for `menu.json` |
| `accessor.py` | Python module — loads `menu.json`, exposes `MENU`, `SEARCH_INDEX`, and `_render_menu` |
| `__init__.py` | Re-exports `MENU` and `_render_menu` as the public `menu` package API |
| `test_accessor.py` | CLI tool — print full menu or fuzzy-search items by keyword |
| `menu_for_system_prompt.txt` | Pre-rendered menu text for embedding in the LLM system prompt |

---

## Python module (`accessor.py`)

`accessor.py` is the runtime interface between `menu.json` and the rest of the application. It exposes two public names, both re-exported by `__init__.py`:

### `MENU`

A module-level `dict[int, dict]` loaded once at import time. Keys are item IDs; each value is a flat dict:

```python
{
    "id":      int,    # globally unique item ID
    "name":    str,    # English name, with preparation type appended when present
                       #   e.g. "Gobi Manchurian (dry)"
    "section": str,    # name of the containing section
    "price":   int,    # price in INR
}
```

Typical usage — look up a single item by ID:

```python
from menu import MENU

item = MENU[42]   # {"id": 42, "name": "...", "section": "...", "price": ...}
```

### `_render_menu(initial_indent=0) -> str`

Returns the full menu as a human-readable indented string, suitable for embedding in an LLM system prompt. Each item is formatted as:

```
<section name>:
 [<id>] <name> — ₹<price>
```

Sub-sections are indented one additional level. Top-level sections are separated by a blank line.

```python
from menu import _render_menu

print(_render_menu())
# Breakfast:
#  [1] Idly — ₹40
#  [2] Vada — ₹35
#  ...

print(_render_menu(initial_indent=4))  # indent everything by 4 spaces
```

## Smoke-testing and search (`test_accessor.py`)

`test_accessor.py` is a CLI tool for two things: printing the full menu and fuzzy-searching items by keyword. Run it directly from the `menu/` directory.

### Print full menu

```bash
cd menu
python test_accessor.py menu
```

Prints all 339 items grouped by section to stdout. Useful to regenerate `menu_for_system_prompt.txt`:

```bash
python test_accessor.py menu > menu_for_system_prompt.txt
```

### Fuzzy search

```bash
python test_accessor.py search <terms>
```

Pass one or more English keywords. The search scores each term against item names and their full ancestor section path, so you can search by item name, section name, ingredient, or preparation type.

```bash
python test_accessor.py search masala dosa
# Found 10 match(es) for 'masala dosa':
#   [30] Masala Dosa (Dosa Treat) — ₹100
#   [34] Paper Massla Dosa (Dosa Treat) — ₹140
#   [36] Butter Masala Dosa (Dosa Treat) — ₹115
#   ...

python test_accessor.py search gobi manchurian
# Found 10 match(es) for 'gobi manchurian':
#   [73] Gobi (dry) (Manchurian / Chilly) — ₹190
#   [74] Gobi (gravy) (Manchurian / Chilly) — ₹195
#   [117] Veg. Ball Manchurian (Manchurian Gravy) — ₹230
#   ...

python test_accessor.py search starters paneer
# Finds paneer items across Tasty Starters sub-sections (Manchurian/Chilly, Pepper Dry, Special Starters)

python test_accessor.py search biryani
# Finds all biriyani variants under Basumathi Bandhar

python test_accessor.py search ice cream chocolate
# Finds chocolate ice cream and sundae variants under Cream Of Parlour / Special Make Ice Creams

python test_accessor.py search coffee
# Found 2 match(es) for 'coffee':
#   [2] Coffee (Hot Beverages) — ₹32
#   [358] Cold Coffee (Milk Shakes) — ₹110
```

Search uses `rapidfuzz` (`token_set_ratio`, threshold 55) if installed, falling back to `difflib`. Results are capped at 10 matches.

## System prompt text (`menu_for_system_prompt.txt`)

`menu_for_system_prompt.txt` is a pre-rendered snapshot of the full menu, ready to paste or `read()` directly into an LLM system prompt. It is generated by running `test_accessor.py` and redirecting stdout:

```bash
python3 menu/test_accessor.py menu > menu/menu_for_system_prompt.txt
```

**Regenerate this file whenever `menu.json` changes** (i.e. after re-running `convert.py`).

### Format

Each top-level section is a heading line followed by indented item lines, with a blank line between sections:

```
<Section name>:
 [<id>] <item name> — ₹<price>

<Next section>:
 [<id>] <item name> — ₹<price>
```

Sub-sections are indented one extra level relative to their parent. Items that have a preparation type carry it in parentheses after the name (e.g. `Gobi Manchurian (dry)`). Item IDs are stable across regenerations as long as `scanned-menu.json` is not reordered — the waiter uses them to reference ordered items unambiguously.

---

The source file has a single top-level `menu` object whose keys are section identifiers. Each section carries its display names, an optional availability window, and either a flat `items` array or a `sub_sections` object containing further sections.

```
{
  "menu": {
    "<SECTION_KEY>": {
      "section_name":          string,            // display name in English
      "section_name_kannada":  string,            // display name in Kannada
      "timings":               string | absent,   // e.g. "8 am to 10 pm"
                                                  //   or "12 noon to 3.30 pm - 7 pm to 10.30 pm"
      "items": [                                  // present when section has no sub-sections
        {
          "name":           string,
          "name_kannada":   string,
          "price":          number,               // regular price (INR)
          "timings":        string | absent       // item-level override
        }
      ],
      "sub_sections": {                           // present instead of items (e.g. TASTY_STARTERS)
        "<SUB_SECTION_KEY>": {
          "sub_section_name":          string,
          "sub_section_name_kannada":  string | absent,
          "items": [
            {
              "name":          string,
              "name_kannada":  string,
              "price":         number,            // used when preparation type is not applicable
              "price_dry":     number,            // used when item has dry variant
              "price_gravy":   number             // used when item has gravy variant
            }
          ]
        }
      }
    }
  }
}
```

### Input notes

- There are 26 top-level sections. Only `TASTY_STARTERS` uses `sub_sections`; all others use `items` directly.
- `timings` is a free-text string and may describe two shifts separated by ` - ` (e.g. `"12 noon to 3.30 pm - 7 pm to 10.30 pm"`).
- Items in `MANCHURIAN_CHILLY` have `price_dry` and `price_gravy` instead of `price`.
- All other items have a single `price`.

## Output structure (`menu.json`)

The output is a fully embedded hierarchy. Every section and item carries a globally unique `id`. Text (display names) is separated into `sectionText` / `itemText` arrays keyed by `langCode`, making it straightforward to add further languages. All field names use camelCase.

```
{
  "languages": [
    { "code": "en", "name": "English" },
    { "code": "kn", "name": "Kannada" }
  ],
  "section": [
    {
      "id":           number,         // globally unique across sections and items
      "parentId":     number | null,  // null for top-level; parent section id for nested
      "sectionKey":   string,         // e.g. "TASTY_STARTERS"
      "name":         string,         // English display name (shortcut — same as sectionText[en])
      "displayOrder": number,
      "startTime":    string,         // "HH:MM" 24-hour; "00:00" when always available
      "endTime":      string,         // "HH:MM" 24-hour; "23:59" when always available
      "sectionText": [
        { "langCode": "en", "name": string },
        { "langCode": "kn", "name": string }
      ],
      "item": [
        {
          "id":              number,
          "name":            string,          // English display name (shortcut — same as itemText[en])
          "displayOrder":    number,
          "preparationType": null | "dry" | "gravy",
          "startTime":       string,
          "endTime":         string,
          "itemText": [
            { "langCode": "en", "name": string },
            { "langCode": "kn", "name": string }
          ],
          "price": number
        }
      ],
      "section": [ ... ]   // nested child sections — same shape, recursively
    }
  ]
}
```

### Output notes

- **Global IDs** — sections and items share one incrementing counter, so `id` is unique across the entire document. Safe to use as a primary key when loading into an RDBMS.
- **`name` shortcut** — both Section and Item carry a top-level `name` field with the English display name, duplicating the `en` entry in `sectionText` / `itemText`. This avoids having to filter the text array for the common case of displaying in English.
- **Self-referencing sections** — child sections (e.g. `MANCHURIAN_CHILLY` inside `TASTY_STARTERS`) are embedded under the parent's `section` array and also carry `parentId`, giving both a tree representation and a foreign-key reference in one.
- **Timings** — free-text timing strings are parsed to `startTime` / `endTime` in 24-hour `HH:MM`. Two-shift timings use the first shift's start and the last shift's end. Missing timings default to `00:00` / `23:59` (always available).
- **Localization** — all display strings live in `sectionText` / `itemText`. Adding a new language requires only inserting additional `{ langCode, name }` entries; the structural fields are language-neutral.

### Item splitting rules

Source items are expanded into multiple output items in two cases:

| Rule | Trigger | Result |
|---|---|---|
| **Slash choice** | `name` contains ` / ` (e.g. `"Coffee / Tea"`) | Split into 2 items — one per variant — sharing the same `displayOrder` and `price`. Kannada name is split on ` /` accordingly. |
| **Dry / gravy** | Item has both `price_dry` and `price_gravy` fields | Split into 2 items with `preparationType: "dry"` and `preparationType: "gravy"`, each with its own price. |

In both cases the resulting items share the same `displayOrder` as their source item.

## Schema (`menu-schema.json`)

`menu-schema.json` is a [JSON Schema draft-07](https://json-schema.org/specification-links#draft-7) description of `menu.json`. It documents every field, its type, and any constraints (e.g. `HH:MM` pattern for time fields, `preparationType` enum).

Validate the output after regeneration using [ajv-cli](https://github.com/ajv-validator/ajv-cli):

```bash
# install once
npm install -g ajv-cli

ajv validate -s menu/menu-schema.json -d menu/menu.json
```

Or with Python ([jsonschema](https://python-jsonschema.readthedocs.io)):

```bash
uv add --dev jsonschema

uv run python -c "
import json, jsonschema
schema = json.load(open('menu/menu-schema.json'))
data   = json.load(open('menu/menu.json'))
jsonschema.validate(data, schema)
print('valid')
"
```

```bash
# Reads scanned-menu.json, writes menu.json (both in this folder)
python menu/convert.py

# Explicit paths
python menu/convert.py <input.json> <output.json>
python menu/convert.py menu/scanned-menu.json menu/menu.json
```
